2. CÁC DATA STRUCTURES QUAN TRỌNG

STACK & QUEUE - O(1) push/pop 

Stack (LIFO): [1,2,3] → push(4) → [1,2,3,4] → pop() → [1,2,3]
Queue (FIFO): [1,2,3] → enqueue(4) → [1,2,3,4] → dequeue() → [2,3,4]

System use: Call stack, BFS queue, message queues

DICTIONARY (Hash Table) - O(1) average

OperationHash TableArrayBST
SearchO(1) avgO(n)O(log n)
InsertO(1) avgO(1) end, O(n) midO(log n)
DeleteO(1) avgO(n)O(log n)

System use: Redis, caching, indexing, dedup

Gotcha: O(1) là average. Tình huống tệ nhất (nhiều collision) là O(n).

BINARY SEARCH TREE - O(log n) 

        8
       / \
      3   10
     / \    \
    1   6    14

Rule: Left child < Parent < Right child

Balanced vs Unbalanced:

Balanced (good):     Unbalanced (bad - degrades to O(n)):
    4                    1
   / \                    \
  2   6                    2
 / \ / \                    \
1  3 5  7                    3
                              \
                               4

System use: Database indexes (B-tree variant), in-memory sorted data

PRIORITY QUEUE (Heap) - O(log n) insert/extract

Luôn biết min (hoặc max) phần tử ngay lập tức.

Min-Heap:
      1
     / \
    3   2
   / \
  7   4

Extract-min: lấy 1 ra, restructure → O(log n)
Insert: thêm vào, bubble up → O(log n)

System use: Task scheduling, Dijkstra algorithm, rate limiting

HASH TABLE DEEP DIVE

Cách hoạt động:

key → hash(key) → index → bucket

"redis" → hash("redis") = 12345 → 12345 % 1000 = 345 → bucket[345]

Collision resolution:

  1. Chaining: Mỗi bucket là linked list
  2. Open addressing: Tìm slot trống tiếp theo

Load factor: n/buckets. Cao quá → nhiều collision → chậm.

5. QUAN TRỌNG: BLOOM FILTER

Bài toán: Check xem phần tử có trong set không? Set có 1 tỷ items.

Hash table: Tốn 1 tỷ × size mỗi item = rất nhiều RAM

Bloom filter:

  • Space: Rất nhỏ (bits, không phải items)
  • False positive: Có thể nói “có” khi thực ra “không”
  • False negative: KHÔNG BAO GIỜ nói “không” khi thực ra “có”

Use case:

  • “URL này đã crawl chưa?” - nếu bloom nói “chưa” → chắc chắn chưa
  • “User này đã thấy ad này chưa?”
  • Cache: check bloom trước khi query DB

BÀI TẬP

Bài 1: Mày có log streaming system, cần:

  • Detect duplicate log entries (by hash)
  • Process logs theo thứ tự nhận (FIFO)
  • Quickly find logs by request_id

Cần những data structures nào?

Bài 2: Rate limiter: mỗi IP chỉ được 100 requests/phút. Cần track request count per IP. Data structure nào? Complexity của mỗi request check?

Bài 3: Task queue với priority (urgent tasks chạy trước). Dùng gì?
