HASHING LÀ GÌ?

Ý tưởng: Biến dữ liệu bất kỳ → số nguyên cố định → dùng số đó làm vị trí lưu trữ.

BướcVí dụ
Dữ liệu gốc“user_12345”
Qua hàm bămhash(“user_12345”) = 847291
Lấy vị trí847291 % 1000 = 291
Lưu vàobucket[291]

Kết quả: Tra cứu O(1) thay vì O(n).

TẠI SAO SYSTEM ENGINEER CẦN BIẾT?

Chỗ dùngHashing làm gì
Redis GET/SETKey → hash → vị trí trong memory
Database index (hash)Column value → hash → row location
Load balancer (consistent hashing)Request → hash → chọn backend server
Dedup (loại bỏ trùng lặp)Content → hash → đã thấy chưa?
Cache keyURL + params → hash → cache entry
Password storagePassword → hash → lưu hash, không lưu plain text
Git commit IDContent → SHA-1 hash → commit identifier

HASH TABLE HOẠT ĐỘNG NHƯ NÀO

Thao tácCách làmĐộ phức tạp
Insert(key, value)hash(key) → vị trí → lưu valueO(1) trung bình
Get(key)hash(key) → vị trí → lấy valueO(1) trung bình
Delete(key)hash(key) → vị trí → xóaO(1) trung bình

Lưu ý: O(1) là trung bình, worst case vẫn O(n) nếu nhiều collision.

COLLISION (VA CHẠM) LÀ GÌ?

Vấn đề: Hai key khác nhau có thể hash ra cùng vị trí.

KeyHashVị trí (mod 10)
“apple”128477
“banana”938477

→ Cả hai muốn vào bucket[7] → Collision!

CÁCH XỬ LÝ COLLISION

Phương phápCách làmƯu/Nhược
Chaining (chuỗi)Mỗi bucket là linked list, collision thì thêm vào listĐơn giản, nhưng tốn memory cho pointers
Open addressingCollision thì tìm slot trống tiếp theoTiết kiệm memory, nhưng phức tạp hơn

Thực tế: Hầu hết implementations dùng chaining.

LOAD FACTOR VÀ PERFORMANCE

Load factor = n / m (n items, m buckets)

Load factorHiệu suấtGiải thích
< 0.5TốtÍt collision
0.5 - 0.75OKChấp nhận được
> 0.75TệNhiều collision, nên resize

Resize: Khi load factor cao → tạo bảng mới lớn hơn → rehash tất cả items.

PHÂN BỐ TRONG HASH TABLE

Khi hash n items vào n buckets (load factor = 1):

Số items trong bucketTỷ lệ buckets
0 (trống)~36.8%
1~36.8%
2~18.4%
3+~8%

Insight: Dù load factor = 1, vẫn có 1/3 buckets trống và một số buckets có nhiều items.

BLOOM FILTER

Bài toán: Kiểm tra “đã thấy item này chưa?” với hàng tỷ items mà không tốn nhiều RAM.

Tiêu chíHash TableBloom Filter
Bộ nhớO(n) - lưu tất cả itemsO(1) - chỉ lưu bits
Trả lời “Có”Chắc chắn cóCó thể sai (false positive)
Trả lời “Không”Chắc chắn khôngChắc chắn không

Cách hoạt động:

BướcInsertQuery
1Hash item với k hàm hash khác nhauHash item với cùng k hàm hash
2Set k bits tương ứng = 1Kiểm tra k bits
3-Tất cả = 1 → “Có thể có”. Có bit = 0 → “Chắc chắn không”

Use cases:

Ứng dụngTại sao Bloom Filter
URL đã crawl chưa?Nếu “không” → chắc chắn chưa, crawl đi
User đã thấy ad này chưa?False positive = show lại ad cũ, chấp nhận được
Cache miss checkCheck bloom trước khi query DB
Spam email filterCheck nhanh trước khi deep scan

CONSISTENT HASHING (QUAN TRỌNG CHO DISTRIBUTED SYSTEMS)

Bài toán: 3 cache servers, hash(key) % 3 chọn server.

KhiVấn đề
Thêm server 4hash(key) % 4 → hầu hết keys đổi server → cache miss hàng loạt
Server 2 chếthash(key) % 2 → tất cả keys đổi server → cache miss hàng loạt

Giải pháp: Consistent Hashing

Ý tưởngChi tiết
Vòng tròn hashServers và keys đều hash lên vòng tròn 0 → 2³²
Chọn serverKey đi theo chiều kim đồng hồ, gặp server nào đầu tiên thì vào đó
Thêm/xóa serverChỉ keys giữa server cũ và server mới bị ảnh hưởng

Kết quả:

Thay đổiHash thườngConsistent Hash
Thêm 1 server (3→4)~75% keys đổi~25% keys đổi
Xóa 1 server (3→2)~67% keys đổi~33% keys đổi

Dùng ở đâu: Redis Cluster, Cassandra, CDN, Load Balancers.

SO SÁNH CÁC PHƯƠNG PHÁP TRA CỨU

Phương phápTra cứuInsertCó thứ tự?Khi nào dùng
Hash tableO(1) avgO(1) avgKhôngCần tra cứu nhanh theo key
BST (cây)O(log n)O(log n)CóCần thứ tự + range query
Sorted arrayO(log n)O(n)CóDữ liệu tĩnh, ít thay đổi
Unsorted arrayO(n)O(1)KhôngHiếm khi tra cứu

HASHING TRONG 4 GIAI ĐOẠN QUẢN LÝ HỆ THỐNG

GIAI ĐOẠN 1: THIẾT KẾ

Bài toán thiết kếHashing giải quyết như nàoVí dụ cụ thể
Phân tải requests giữa nhiều serversConsistent hashing quyết định request đi server nào5 API servers, request từ user_123 luôn đi server 2
Cache distributedHash key để biết cache ở server nàoSession của user_456 lưu ở Redis node 3
Database shardingHash column để biết row ở shard nàoOrder #789 lưu ở DB shard 1
Dedup trong pipelineHash content để phát hiện trùngLog line đã xử lý chưa?
Idempotency keyHash request để tránh xử lý 2 lầnPayment request bị retry, không charge 2 lần

Câu hỏi thiết kế hay gặp:

  • “Làm sao scale cache lên 10 servers mà không mất hết data khi thêm/bớt server?” → Consistent hashing
  • “Làm sao đảm bảo cùng 1 user luôn đi cùng 1 server (sticky session)?” → Hash user_id
  • “Database có 100 triệu rows, làm sao chia ra nhiều servers?” → Hash sharding key

GIAI ĐOẠN 2: TRIỂN KHAI (DEPLOY)

Tình huống deployHash liên quan như nàoHậu quả nếu không hiểu
Thêm Redis node vào clusterConsistent hash redistribute keysKhông hiểu → tưởng data mất
Scale up cache servers 3→5Keys migrate giữa nodesKhông hiểu → cache miss spike, DB overload
Deploy DB shard mớiCần resharding strategyKhông hiểu → downtime kéo dài
Rollout load balancer mớiHash algorithm có thể khácKhông hiểu → sticky session break

Checklist khi deploy liên quan đến hashing:

Kiểm traTại sao
Hash algorithm có thay đổi không?Đổi algorithm = tất cả keys đổi vị trí
Số nodes có thay đổi không?Thay đổi = một phần keys migrate
Có warmup cache trước không?Cache miss sau deploy = DB spike

GIAI ĐOẠN 3: VẬN HÀNH

Hiện tượng vận hànhNguyên nhân liên quan hashCách kiểm tra
1 cache server CPU cao, còn lại idleHash không đều (hot key)Check key distribution
Cache miss rate tăng đột ngộtCó server mới join/leave clusterCheck cluster membership
Database 1 shard overloadHash function không phân bố đềuCheck row count per shard
Request từ cùng user đi nhiều servers khác nhauLoad balancer không dùng consistent hashCheck LB config

Metrics cần monitor:

MetricÝ nghĩaAlert khi
Key distribution per nodeCó đều không?1 node có >150% avg
Cache hit rateHash có hiệu quả không?Drop >10%
Shard sizeDB sharding có đều không?1 shard >200% avg

GIAI ĐOẠN 4: XỬ LÝ SỰ CỐ (TROUBLESHOOTING)

Sự cốTriệu chứngNguyên nhân hashGiải pháp
Cache thundering herdDB bị spike requests đột ngộtNhiều keys expire cùng lúc hoặc cache server chếtStaggered TTL, consistent hash với replicas
Hot partition1 DB shard chậm, còn lại OKHash key không đều, hoặc 1 key quá hotThêm salt vào hash, hoặc tách hot key riêng
Session loss sau deployUsers bị logout hàng loạtHash algorithm thay đổi, session đổi serverRollback, hoặc dùng centralized session store
Data inconsistency giữa replicasRead từ khác servers ra khác kết quảHash ring chưa sync sau node join/leaveWait for rebalancing complete

Quy trình debug hash-related issues:

BướcHành độngTool/Command
1Check cluster membership có thay đổi gần đây?redis-cli cluster nodes, logs
2Check key distribution có đều không?redis-cli --cluster info
3Check hot keysredis-cli --hotkeys
4Check hash algorithm configLB config, app config

CONSISTENT HASHING - ĐI SÂU HƠN

Tại sao quan trọng: Đây là foundation của hầu hết distributed cache/database.

So sánh: Hash thường vs Consistent Hash

Tiêu chíhash(key) % n serversConsistent hashing
Khi thêm 1 server (3→4)~75% keys đổi server~25% keys đổi
Khi mất 1 server (3→2)~67% keys đổi server~33% keys đổi
Cache miss sau thay đổiMassive, DB overloadMinimal, chấp nhận được
ComplexityĐơn giảnPhức tạp hơn chút

Consistent Hashing hoạt động như nào (không code)

BướcGiải thích
1. Vòng tròn hashTưởng tượng vòng tròn từ 0 đến 2³²
2. Đặt servers lên vòngServer A ở vị trí hash(“A”), Server B ở vị trí hash(“B”)…
3. Key tìm serverKey hash ra vị trí, đi theo chiều kim đồng hồ, gặp server nào đầu tiên thì vào đó
4. Thêm serverServer mới chen vào vòng, chỉ keys giữa nó và server trước bị ảnh hưởng
5. Xóa serverKeys của server đó chuyển sang server kế tiếp trên vòng

Virtual nodes (vnodes) - Giải quyết phân bố không đều

Vấn đềGiải pháp
3 servers có thể phân bố không đều trên vòngMỗi server có 100-200 virtual nodes rải đều
Server mạnh hơn nên nhận nhiều traffic hơnServer mạnh có nhiều vnodes hơn

BLOOM FILTER - KHI NÀO DÙNG

Tình huốngDùng Bloom Filter?Lý do
Check username tồn tại khi đăng ký✗ KhôngCần chính xác 100%, false positive = cho đăng ký trùng
Check URL đã crawl chưa✓ CóFalse positive = skip URL, chấp nhận được
Check email có trong spam list✓ CóFalse positive = check thêm, không spam lọt
Check user đã xem ad chưa✓ CóFalse positive = show lại ad cũ, không sao
Cache lookup optimization✓ CóCheck bloom trước, negative = skip DB query

Quy tắc: Dùng Bloom Filter khi false positive chấp nhận được và cần tiết kiệm memory.
