DYNAMIC PROGRAMMING LÀ CÁI GÌ?

Một câu: Giải bài toán lớn bằng cách lưu lại kết quả của các bài toán nhỏ hơn, tránh tính lại nhiều lần.

Nói cách khác: Đánh đổi bộ nhớ lấy tốc độ.

VÍ DỤ ĐƠN GIẢN NHẤT - DÃY FIBONACCI

CáchFibonacci(5) tính như nàoSố lần tính Fib(2)
Đệ quy thườngFib(5) = Fib(4) + Fib(3), rồi Fib(4) = Fib(3) + Fib(2)… cứ thế3 lần (tính lại!)
DPTính Fib(1), Fib(2), lưu lại. Dùng để tính Fib(3), lưu lại. Dùng để tính Fib(4)…1 lần (lưu rồi dùng lại)
CáchĐộ phức tạpFib(50) mất bao lâu
Đệ quy thườngO(2ⁿ)Hàng triệu năm
DPO(n)Microseconds

KHI NÀO BÀI TOÁN DÙNG ĐƯỢC DP?

Điều kiệnNghĩa là gìVí dụ
Optimal substructureLời giải tối ưu của bài toán lớn = tổ hợp lời giải tối ưu của bài toán nhỏĐường ngắn nhất A→C = đường ngắn nhất A→B + đường ngắn nhất B→C
Overlapping subproblemsCùng bài toán nhỏ bị tính đi tính lại nhiều lầnFib(3) được gọi nhiều lần khi tính Fib(5)

Nếu có cả 2 điều kiện → DP sẽ giúp.

DP TRONG SYSTEM ENGINEERING

Bài toánDP giải như nào
Resource allocationChia X CPU cores cho N services, maximize throughput
Capacity planningVới budget B, chọn combo servers nào tối ưu nhất
Batch job schedulingChọn jobs nào chạy để maximize value trong time window
Cache optimizationVới cache size C, giữ items nào để maximize hit rate
Network packet routingTìm đường với constraint bandwidth/latency
Log retention policyGiữ logs nào trong storage limit S

BÀI TOÁN KINH ĐIỂN: KNAPSACK (BA LÔ)

Tình huống system: Mày có 100GB storage, cần chọn logs/backups nào giữ lại.

ItemSizeValue (importance)
App logs tuần này30GB100
App logs tuần trước30GB60
DB backup hôm nay50GB90
DB backup hôm qua50GB40
System metrics20GB50

Câu hỏi: Chọn combo nào để tổng size ≤ 100GB và tổng value cao nhất?

CÁCH GIẢI KHÔNG DP (BRUTE FORCE)

CáchSố combo phải thửVới 20 items
Thử tất cả2ⁿ2²⁰ = 1 triệu combo
Với 50 items2⁵⁰1 triệu tỷ combo (vỡ)

CÁCH GIẢI DP

Ý tưởng: Xây bảng từ bài toán nhỏ → lớn.

SubproblemNghĩa
dp[i][w]Value tối đa khi xét i items đầu tiên, với capacity w

Xây bảng:

Items xétCapacity 020GB30GB50GB80GB100GB
Không item nào000000
+ App logs tuần này (30GB, 100)00100100100100
+ App logs tuần trước (30GB, 60)00100100160160
+ DB backup hôm nay (50GB, 90)00100100160190
…………………

Kết quả: Ô cuối cùng = value tối đa có thể đạt được.

ĐỘ PHỨC TẠP

CáchThời gianSpace
Brute forceO(2ⁿ)O(1)
DPO(n × W)O(n × W)

Với n=50 items, W=100GB: DP = 5000 operations. Brute force = 1 triệu tỷ.

CÁC BÀI TOÁN DP KHÁC TRONG SYSTEMS

1. Shortest Path (Dijkstra cũng là DP!)

SubproblemNghĩa
dist[v]Khoảng cách ngắn nhất từ source tới v
Cập nhậtdist[v] = min(dist[v], dist[u] + weight(u,v))

2. Job Scheduling với Deadlines

JobDurationDeadlineProfit
A2h4h100
B1h2h50
C3h6h120

Câu hỏi: Chọn jobs nào để maximize profit, mỗi job phải xong trước deadline.

3. Resource Allocation

ScenarioDP formulation
N servers, mỗi cái có cost và capacityChọn combo maximize capacity với budget B
M tasks, N workersAssign tasks để minimize total time

DP VS GREEDY

Tiêu chíGreedyDP
Cách chọnChọn cái tốt nhất LÚC NÀYXét TẤT CẢ khả năng qua bảng
Có luôn tối ưu?Không, có thể saiCó, nếu formulate đúng
Tốc độThường nhanh hơnChậm hơn nhưng chính xác
Dùng khiBài toán có greedy propertyBài toán tối ưu hóa tổng quát

Ví dụ Greedy sai:

ItemSizeValueValue/Size
A10606
B201005
C301204

Capacity = 30. Greedy (chọn value/size cao nhất) → chọn A (10) → còn 20 → không vừa B hay C → Total = 60.

Tối ưu thực sự → chọn C → Total = 120.

KHI NÀO KHÔNG CẦN DP

Tình huốngDùng gì thay
Bài toán nhỏ (n < 20)Brute force đủ nhanh
Có greedy propertyGreedy đơn giản hơn
Không có overlapping subproblemsDivide & conquer
Real-time constraintHeuristics, approximate

THỰC TẾ: DP TRONG CÔNG VIỆC HÀNG NGÀY

Mày có thể không viết DP, nhưng…DP chạy bên trong
Dùng OSPFDijkstra = DP
Query optimizer của databaseChọn join order tối ưu = DP
Compiler optimizationRegister allocation = DP
Kubernetes schedulerBin packing = DP variant

Điều kiệnCheck

GreedyDP
